Schema V1 in plain English

actions.json is a map for agents.

It tells an agent what a website intentionally allows, where those abilities live on the page, and how to know when the map has gone stale.

What can I do?Declared actions and events
Where is it?Page targets and source hints
Is it safe now?State, checks, and trust

The core promise

A website should not force every agent to rediscover the same buttons, forms, carousels, and safe paths from scratch. The site, the user, or a trusted storage source can publish a readable map instead.

Metaphor

A map is more than a list of roads.

The schema describes more than tools. It maps places, conditions, routes, warnings, and repair checks for a living website.

Tools

Things the agent may ask the website to do, like submit a form or open an overlay.

Think of these as approved verbs. A transition may name the tool to call.

Signals

Things the page tells the agent, like “the user clicked this launcher.”

These are page events turned into structured context, not free-form chat.

Checks

Questions the runtime can ask: “Does this selector still exist?” “Is this the right page?”

Checks catch drift before an action touches the wrong element.

States

Named situations, like “carousel visible” or “form ready.”

State names explain what is true before an action is available.

Transitions

Edges between states, like “visible carousel” to “scrolled carousel.”

A transition is not a tool. It explains the before state, after state, tool to call, limits, and stop rules.

Attachments

Page additions, like a button that reopens an agent-created overlay.

Attachments are visible add-ons that can be reinstalled after navigation.
Root shape

The top-level file says what kind of map this is.

Each root field answers one plain question. You do not need to read every advanced section to understand a small map.

{ "protocol": "actions.json" }

Protocol tells the runtime which manifest standard this file follows.

{ "version": 1 }

Version tells validators which schema rules to apply.

{ "surface": { ... } }

Surface names the origin, page kind, and human-readable context for the map.

{ "imports": [ ... ] }

Imports compose website, private, shared, and public maps under explicit trust rules.

{ "prompt": { ... } }

Prompt gives the agent page-specific instructions without hiding authority.

{ "context": [ ... ] }

Context is scoped documentation loaded as the agent moves through pages, states, targets, and tasks.

{ "states": [ ... ] }

States are named facts such as authorized, issue visible, carousel at end, or form ready.

{ "transitions": [ ... ] }

Transitions connect states and may name the tool to call, plus waits, rate limits, and stop conditions.

{ "tools": [ ... ] }

Tools are declared calls the agent may ask the runtime to execute.

{ "signals": [ ... ] }

Signals are page-originated events that can become structured agent context.

{ "attachments": [ ... ] }

Attachments describe runtime-installed UI, like overlay launchers that reattach after navigation.

{ "checks": [ ... ] }

Checks probe selectors, state, and execution paths so drift is caught before action.

{ "provenance": { ... } }

Provenance records where the map came from and when it was updated.

Agent context

The map also teaches the agent where it is.

Some of actions.json is not a button or a handler. It is plain-English context that gets loaded when the agent reaches the right page, state, target, or task.

Where am I?

Explain the current website area.

What is true?

Name the current state before action.

What is available?

List relevant tools, signals, and targets.

When to load?

Match URL, state, target, or attachment predicates.

What to inspect?

Point to diagnostics and observables.

What not to assume?

Documentation only; authority comes from tools.

Plain English: prompt is broad surface guidance. context[] is scoped guidance that follows the agent as it traverses the website.
Targets vs sources

“Where on the page?” is different from “where in the code?”

This split is central. A target points at the live website. A source points at the implementation or evidence.

target

The live page location: selector, role, visible text, URL condition, current state, and fallback selectors.

source

The files, symbols, selectors, or component names that help a human or agent debug the map.

Execution

A handler is convenient, but steps make behavior inspectable.

A handler is page code that does the job. Execution steps are the intended readable fallback or test trace: inspect, type, click, wait, extract.

Handler first

Call the existing page function. Use steps only for validation, tracing, or fallback.

Steps first

Implementation pending. The schema can describe this path before the runtime can execute it generically.

Documentary

Steps explain what should happen but are not runtime authority.

Test only

Steps are used by checks to verify that the map still works.

Plain English: this is the difference between “ask the app to submit the form” and “find the input, type the name, click Submit, then read the result.”
Checks and drift

The map must admit when the road moved.

A living website changes. Checks are probes that catch broken selectors, wrong contexts, missing buttons, or unsafe states before the agent acts.

Info

Non-blocking evidence. Useful for learning.

Minor

Small selector or copy drift with low risk.

Major

The action likely fails or appears in the wrong place.

Critical

The action could touch the wrong private data, purchase, payment, or destructive surface.

Evidence

Store URL, screenshot, DOM snapshot, and matched selectors when a check fails.

Contingency

Try alternate selectors or hand off to the user instead of guessing.

Imports and trust

One user may combine many maps.

A website can publish a map. A user can have private memory. A group can share additions. A public package can contribute generic knowledge. The schema needs rules for combining them.

Namespace

A prefix that keeps imported names from colliding, like private.linear.

Trust

Where this map came from: website, private, shared, public, local, or unknown.

Override policy

Which source wins when two maps describe the same action.

Privacy boundary

Public maps should not receive private browsing data unless a runtime explicitly grants it.

Signals and protocol

Tools are agent requests. Signals are page events.

The schema keeps these separate, then explains how both travel through a bridge using Responses-style item shapes.

action_call

The agent asks the runtime to run a declared tool.

action_call_output

The runtime returns the structured result for that call.

dom_event

The runtime forwards a validated page signal.

action_error

The runtime returns a stable error code with evidence.

call_id

A tracking id that ties a request to its result or error.

runtime_id

A tracking id for the browser surface, such as an authorized tab.

Review checklist

How to review the schema without getting lost.

Read from the general idea to the concrete example. If the simple form still feels readable, the schema is probably reviewable.

1

Start with the purpose and design principles. Confirm that “map, not just tool list” feels right.

2

Review root fields, scoped context, target descriptors, states, and attachments. These are the biggest conceptual additions.

3

Review execution steps and checks. Decide whether these belong inside the manifest or beside it.

4

Review imports and trust. This connects directly to private, shared, and public storage scopes.

5

Read the bridge protocol appendix. It should be specific enough to implement, but not locked to one provider.

6

End with examples/simple-form/actions.json. Use it as the sanity check for readability.
